# 이력서/자기소개서 작성 가이드라인

## 개요

이력서와 자기소개서 작성 시 적절한 톤과 표현을 사용하여 역량을 효과적으로 어필하되, 과장되거나 공격적인 표현을 피하는 가이드라인을 제공한다.

## 핵심 원칙

### 1. 겸손하되 명확하게

- 성과를 숨기지 않되, 과장하지 않음
- 구체적인 수치와 맥락을 함께 제공
- "~했습니다"보다 "~하여 ~의 결과를 얻었습니다" 형식 권장

### 2. 맥락 중심의 표현

성과의 크기보다 **문제 해결 과정**과 **비즈니스 가치**를 강조한다.

**피해야 할 표현:**
- "1000배 개선" → 실제로는 잘못된 설계를 바로잡은 것
- "혁신적인 아키텍처 설계" → 구체적이지 않음
- "대규모 시스템 구축" → 정량화 없음

**권장 표현:**
- "기존 설계의 병목 지점을 분석하고 개선하여 응답 시간 단축"
- "서비스 안정성 향상을 위해 모니터링 체계 구축"
- "팀 생산성 향상을 위한 CI/CD 파이프라인 도입"

### 3. STAR 기법 적용

경력 기술 시 다음 구조를 따른다:

| 요소 | 설명 | 예시 |
|------|------|------|
| **S**ituation | 배경/상황 | "레거시 시스템의 배포 프로세스가 수동으로 진행되어..." |
| **T**ask | 과제/목표 | "자동화된 배포 파이프라인 구축이 필요했습니다" |
| **A**ction | 주요 작업/역할 | "Github Actions 기반 CI/CD 파이프라인을 설계 및 구축했습니다" |
| **R**esult | 성과/결과 | "배포 시간이 2시간에서 15분으로 단축되었습니다" |

### 4. 수치 표현 가이드

**적절한 수치 표현:**
- 비율 개선: "40% 절감", "70% 향상"
- 시간 단축: "2시간 → 15분"
- 규모: "일일 10만 요청 처리"

**피해야 할 표현:**
- 극단적 배수: "1000배 개선", "무한대로 확장"
- 모호한 수치: "대폭 개선", "크게 향상"
- 검증 불가능한 수치: "업계 최고 수준"

### 5. 역할 표현 가이드

**적절한 역할 표현:**
- "설계 및 구현을 담당했습니다"
- "팀 리드로서 방향성을 제시했습니다"
- "핵심 API 50% 이상을 개발했습니다"

**피해야 할 표현:**
- "전적으로 책임졌습니다" (팀 기여 무시)
- "혼자서 구축했습니다" (협업 부정)
- "모든 것을 결정했습니다" (독단적 인상)

## 문서별 가이드

### 자기소개서 (Cover Letter)

**구조:** 3문단 권장

1. **도입부**: 지원 동기 또는 핵심 역량
2. **본문**: 관련 경험과 성과 (JD 키워드 연결)
3. **마무리**: 입사 후 기여 방향

**톤앤매너:**
- 진정성 있는 표현 사용
- 회사/직무에 대한 이해 표현
- 구체적 경험 중심 서술

### 이력서 (Resume)

**구조:**
- 기본 정보 (연락처, 학력)
- 기술 스택
- 경력 사항 (STAR 기법)
- 프로젝트 경험
- 자격증/기타

**경력 기술 예시:**

```markdown
## 위데이터랩 (2023.11 ~ 2025.07)

**역할**: 백엔드 개발 리드

### 치과 진료 시스템 백엔드 개발
- **배경**: 신규 서비스 런칭을 위한 백엔드 시스템 구축 필요
- **역할**: 결제 연동, 치료 프로세스 등 핵심 API 설계 및 구현
- **성과**: 핵심 API 50% 이상 담당, 서비스 정상 런칭에 기여
```

## 표현 변환 예시

| 과한 표현 | 적절한 표현 |
|----------|------------|
| 이미지 로드 속도를 1000배 개선 | 이미지 최적화로 로드 시간을 13초에서 밀리초 단위로 개선 |
| 혁신적인 아키텍처 설계 | 서버리스 아키텍처를 도입하여 운영 효율성 향상 |
| 압도적인 성능 개선 | 응답 시간 40% 단축 |
| 완벽한 시스템 구축 | 안정적인 시스템 운영 기반 마련 |
| 모든 문제를 해결 | 주요 병목 지점을 분석하고 개선 |

## 검증 체크리스트

작성 완료 후 다음 항목을 점검한다:

- [ ] 과장된 수치 표현이 없는가?
- [ ] 모든 성과에 맥락(배경)이 포함되어 있는가?
- [ ] STAR 기법 구조를 따르고 있는가?
- [ ] 팀 기여를 적절히 인정하고 있는가?
- [ ] JD의 핵심 키워드와 연결되어 있는가?
- [ ] 읽는 사람이 이해할 수 있는 구체적 표현인가?
